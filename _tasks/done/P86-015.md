# P86-015: ECS image URI resolution (BUG-693 fix)

**Status:** Done
**Branch:** `phase86-complete-runner-support`

## What

Fixed BUG-693: ECS task definitions were registered with unqualified image references (e.g. `alpine`). Fargate cannot pull bare refs and the task stayed PENDING forever. Ported the Lambda backend's ECR pull-through-cache resolver to ECS; `ContainerCreate` now rewrites short references into `<account>.dkr.ecr.<region>.amazonaws.com/docker-hub/library/alpine:latest`-style URIs before the container record is stored, so `buildContainerDef` sees an ECR-pullable ref.

## Files

| File | Change |
|------|--------|
| `backends/ecs/image_resolve.go` | New. `resolveImageURI`, `ensurePullThroughCache`, `parseDockerRef`, `extractAccountID`. Mirrors Lambda's helper with these differences: reads account from `config.ExecutionRoleARN` (not `RoleARN`); drops the `pushToECR` fallback; degrades to the raw ref on any AWS failure so Fargate surfaces the pull error visibly instead of faking success |
| `backends/ecs/backend_impl.go` | `ContainerCreate` calls `s.resolveImageURI` between image-config merging and container-record construction; result overwrites `config.Image` |
| `backends/ecs/image_resolve_test.go` | New. 6 cases for `parseDockerRef` (alpine / alpine:latest / node:20 / myorg/app / ghcr.io/ / registry:port/) + 3 cases for `extractAccountID` |

## What this enables

Once Fargate gets a real pull-through-cache URI, `docker run alpine ...` flows end-to-end: task def registers, Fargate pulls from ECR (ECR auto-fetches from Docker Hub on first access), task runs to completion.

Already-qualified ECR URIs pass through unchanged, so no regression for workflows that already use `<account>.dkr.ecr...` refs.

## Follow-ups (out of scope)
