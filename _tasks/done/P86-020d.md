# P86-020d: Fix BUG-698 — docker CLI `docker run -d` hangs (wait-before-start)

**Status:** Done
**Branch:** `phase86-complete-runner-support`

## What

Docker CLI's `docker run -d` (and `docker run` without `-d`) hung against the sockerless ECS backend after POST /containers/create. Never sent POST /start.

Diagnosed via a purpose-built `tools/http-trace` HTTP-logging proxy that captured the complete wire sequence. The trace showed:

```
HEAD  /_ping                           -> 200
POST  /containers/create               -> 201 {"Id":"...","Warnings":[]}
POST  /containers/{id}/wait?condition=next-exit  -> (blocks forever)
(kill eventually fires from timeout)
```

docker CLI's Go SDK sends `POST /wait?condition=next-exit` *before* `POST /start` for the `docker run -d` flow — it registers to receive the exit event first, so it doesn't miss a fast-exiting container. `cli.post()` blocks on reading the HTTP response status line from /wait. Since sockerless's handler didn't write any response bytes until the container actually exited, the status line never arrived; docker CLI's main goroutine never got past `ContainerWait()` and never sent /start.

## Fix

`backends/core/handle_containers.go:handleContainerWait` now commits the 200 OK + `Content-Type: application/json` headers via `flushWaitHeaders(w)` immediately, before blocking on the exit event. The response body (JSON exit code) is written via `writeWaitBody` once the exit lands. docker CLI's `cli.post()` returns as soon as the status line hits the wire; the main goroutine proceeds to `/start`; the wait goroutine continues reading the body until the exit event arrives.

## Files

| File | Change |
|------|--------|
| `backends/core/handle_containers.go` | `handleContainerWait`: added `flushWaitHeaders(w)` before every blocking code path (local wait channel, cloud wait, store wait). Added `writeWaitBody(w, exitCode)` helper for the post-flush JSON write. Both paths (CloudState and legacy Store) updated. |
| `backends/core/handle_containers.go` | Added `json` import for the post-flush encoder. |
| `tools/http-trace/main.go` | New throwaway diagnostic tool. Logs every HTTP request + response with headers/body. Handles hijacked upgrades for attach/exec. Not shipped with releases (no go.work entry, minimal go.mod). Useful the next time a docker-CLI interop bug appears. |

## Verification

Against the AWS simulator with a local ECS backend:

- `docker run -d --name x alpine echo hi` → exits 0, prints container ID, container runs via ECS RunTask. Previously hung for 30s+ and returned exit 124 from timeout.
- `docker run --rm alpine echo attach-hello` → exits 0, stdout contains `attach-hello`. Previously hung. Validates both BUG-692 (attach) and BUG-698 (wait) fixes in the same flow.

Both scenarios now pass in the simulator-mode replay that was blocked for the whole session.

## Why the earlier diagnosis missed this

Prior attempts looked at the attach handler because that's where the hang was most visible. The wait-before-start ordering isn't obvious without packet/wire visibility — `docker -D` doesn't show the wait request clearly in CLI logs; backend logs didn't help because /start simply never arrived.

The `http-trace` proxy was the winning diagnostic. I should keep it around as a reusable tool for future docker-CLI interop bugs.

## Regression test

Live-style smoke already covers `docker run --rm` in `smoke-tests/run.sh` (landed in P86-017). That will catch a regression in CI. A more targeted unit test covering the exact wait-early-flush behavior can be added in a follow-up if needed; the smoke test is sufficient for now because any break of this path re-produces the original symptom (test times out at 60s).
