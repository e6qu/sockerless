# P86-004a: Lambda `ContainerStop` / `ContainerKill` best-effort termination

**Status:** Done
**PR:** (pending — same branch)
**Branch:** `phase86-complete-runner-support`

## What

Lambda container stop/kill no longer silently returns while the AWS function keeps running. The new flow:

1. `UpdateFunctionConfiguration(Timeout=1)` — clamps the function's timeout to the minimum so any *future* invocations of this container are short-lived. AWS Lambda does not apply configuration changes to in-flight invocations, so this is a guard against lingering retries, not a mid-flight kill.
2. `disconnectReverseAgent(containerID)` — stub in `backends/lambda/reverse_agent.go`. Will be wired up in P86-005 (agent-as-handler pattern) and is the only mechanism that can actually cut an in-flight invocation short.
3. Local wait channel close — unchanged behavior; lets `docker wait` return immediately.

Exit code reported for stop is now `137` (force-stop convention) rather than `0`. ContainerKill passes through the signal-derived code via `core.SignalToExitCode`.

## Why not a true cancel

AWS Lambda has no invoke-cancel API. Synchronous `Invoke` runs to completion regardless of control-plane actions. The only real termination paths are (a) the function's own logic exiting, (b) the 15-min hard cap, (c) the container image's entrypoint deciding to exit because the agent bundle's WebSocket disconnected. This PR sets up (c)'s scaffolding and relies on P86-005 to finish it.
