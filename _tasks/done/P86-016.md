# P86-016: ECS ContainerAttach via CloudWatch streaming (BUG-692 fix)

**Status:** Done
**Branch:** `phase86-complete-runner-support`

## What

Fixed BUG-692: `docker run --rm alpine echo x` hung against the live ECS backend. Root cause: `ContainerAttach` delegated to `BaseServer.ContainerAttach`, which created a pipe-backed `io.ReadWriteCloser` whose goroutine called `LocalStreamDriver.Attach` — that driver unconditionally returned `"no stream driver connected"`, which closed the pipe, which EOFed the hijacked TCP connection, which left docker CLI spinning.

Now ECS has its own `ContainerAttach` that streams CloudWatch log events as Docker multiplexed-frame bytes (or raw bytes for TTY containers). The stream closes when the ECS task transitions to STOPPED via `core.StreamCloudLogs`'s existing follow-mode container-stopped detection at `backends/core/log_driver.go:92`. Writes are discarded — Fargate tasks expose no remote stdin channel.

The fetch closure previously embedded in `ContainerLogs` is now a package-level helper `buildCloudWatchFetcher(containerID)` used by both `ContainerLogs` and `ContainerAttach`.

## Caveats

- The follow loop polls CloudWatch every 1s via `StreamCloudLogs`. Attach output will lag real-time by up to 1s. Matches `docker logs -f` behavior.
- `attach-before-start` (docker's default `run` flow): the fetch closure returns empty entries when `getTaskID` reports `unknown`; `StreamCloudLogs` retries on each follow tick, so once the task starts and logs begin appearing the stream picks them up.
- TTY detection uses `c.Config.Tty` resolved at attach time. If the container is re-inspected later the TTY choice doesn't update — consistent with docker daemon behavior.
