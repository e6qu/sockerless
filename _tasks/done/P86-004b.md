# P86-004b: Lambda `docker logs -f` lazy stream resolution

**Status:** Done
**PR:** (pending — same branch)
**Branch:** `phase86-complete-runner-support`

## What

Fixed a latent bug where `ContainerLogs` on a Lambda-backed container with `follow=true` would return an empty, never-closing stream if called before the Lambda function's CloudWatch log stream existed.

Lambda creates a new log stream (`/aws/lambda/<function>`) only when the function is first invoked. `ContainerLogs` previously resolved the log stream name *once*, up-front. If the caller opened the log stream before the invocation produced output (common for `docker run` + `docker logs -f`), the cached stream name was nil, the fetch closure returned empty entries, and the follow loop polled empty forever.

## Fix

`ContainerLogs` in `backends/lambda/backend_impl.go` now captures only the log *group* name up-front. The fetch closure calls `resolveStream()` lazily on each invocation, caching the stream name once it appears so we don't pay `DescribeLogStreams` on every poll tick.
