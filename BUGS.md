| 700 | Med | Backend's `cloudNamespaceCreate` fails silently when the underlying subnet resolution fails, so `docker network create` returns success but Cloud Map namespace isn't set up. Subsequent containers fail to register, cross-container DNS doesn't work, and the failure is only visible in backend logs. Should either fail the network create or at minimum surface the warning to the client via a response `Warning` field. | open |
| 701 | High | AWS simulator runs each ECS task as a standalone host-Docker container; when two tasks share a sockerless-network, they don't share a real Docker user-defined network, so Docker's embedded DNS can't resolve cross-task hostnames. Cloud Map registration succeeds but the simulator's tasks can't actually reach each other by name. For parity with real Fargate + Cloud Map, the simulator needs to either (a) create a real docker network per sockerless-network and launch tasks on it, or (b) inject `/etc/hosts` entries + a mini DNS resolver into launched containers that consults Cloud Map. Simulator parity gap — blocks cross-container services DNS testing in simulator mode. | open |
| 699 | Med | AWS simulator's EC2 API didn't pre-register `subnet-sim` (the conventional placeholder subnet used by `SOCKERLESS_ECS_SUBNETS=subnet-sim` in the smoke-test harness). Sockerless's `cloudNetworkCreate` + `cloudNamespaceCreate` called `DescribeSubnets` → returned empty → VPC-ID resolution failed → Cloud Map namespace never got made. Fixed in `simulators/aws/ec2.go` by auto-creating `vpc-sim` + `subnet-sim` entries on simulator startup (idempotent). | fixed |
| 698 | Critical | Docker CLI's `docker run -d` flow sends POST /wait?condition=next-exit *before* POST /start, and blocks on reading the wait response's status line. Sockerless's wait handler called `CloudState.WaitForExit` (or the Store-based wait) *before* writing any response, so Go's http server never sent headers, so docker CLI's `cli.post` never returned, so /start was never sent. Fixed by adding an early `flushWaitHeaders` call that commits the 200 + Content-Type before the handler blocks on the exit event; the response body (exit code JSON) is written after the exit lands. Verified: `docker run -d` and `docker run --rm` both succeed end-to-end against the simulator. | fixed |
